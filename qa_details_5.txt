# 부가세 참고자료 매출 상세 테스트 케이스

## API 정보
- **End Point**: `/api/v1/customer/{customerNo}/additional-tax/sales/detail`
- **Path Parameters**: customerNo (거래처번호)
- **Request Parameters**: startMonth, endMonth, excludeMeet

## 테스트 케이스

### 테스트 케이스 1: 기본 조회 기능 검증
**설명**: 기본적인 부가세 참고자료 매출 상세 조회 기능이 정상적으로 동작하는지 확인
**선행 조건**: 유효한 거래처번호가 존재함
**테스트 단계**:
1. 유효한 거래처번호로 API 호출
2. 필수 파라미터인 startMonth와 endMonth 설정 (예: 2024-01, 2024-03)
3. excludeMeet 파라미터는 기본값(false) 사용

**예상 결과**:
- 상태 코드 200 OK 반환
- 해당 기간의 매출 상세 데이터가 정상적으로 조회됨
- 응답에 txDate, salesType, salesTypeName, payMethod, payMethodText, serviceTxNo, supplyAmount, vatAmount, totalAmount 필드가 포함됨

### 테스트 케이스 2: 배민 부담 가게배달팁 포함 조회
**설명**: 배민 부담 가게배달팁(HVA 배달팁 할인)이 포함된 매출 상세 조회 기능 검증
**선행 조건**: 배민 부담 가게배달팁이 발생한 거래가 존재함
**테스트 단계**:
1. 배민 부담 가게배달팁이 발생한 거래가 있는 거래처번호로 API 호출
2. 해당 거래가 포함된 기간으로 startMonth와 endMonth 설정
3. excludeMeet 파라미터는 기본값(false) 사용

**예상 결과**:
- 상태 코드 200 OK 반환
- 응답 데이터 중 배민 부담 가게배달팁 항목이 포함됨
- 해당 항목의 payMethod가 'BARO_WOOWABROS_CHARGE_DELIVERY_TIP'으로 표시됨
- 해당 항목의 payMethodText가 '배민 부담 가게배달팁'으로 표시됨
- 해당 항목의 salesType이 'PG'로 표시됨
- 해당 항목의 salesTypeName이 '기타매출'로 표시됨
- 금액 정보(supplyAmount, vatAmount, totalAmount)가 정확히 표시됨

### 테스트 케이스 3: 만나서결제 제외 조회
**설명**: excludeMeet 파라미터를 true로 설정하여 만나서결제를 제외한 매출 상세 조회 기능 검증
**선행 조건**: 만나서결제 거래가 존재함
**테스트 단계**:
1. 만나서결제 거래가 있는 거래처번호로 API 호출
2. 해당 거래가 포함된 기간으로 startMonth와 endMonth 설정
3. excludeMeet 파라미터를 true로 설정

**예상 결과**:
- 상태 코드 200 OK 반환
- 응답 데이터에 만나서결제 관련 항목이 제외됨
- 배민 부담 가게배달팁을 포함한 다른 결제 방식의 데이터는 정상적으로 조회됨

### 테스트 케이스 4: 조회 기간 유효성 검증
**설명**: 유효하지 않은 조회 기간으로 API 호출 시 적절한 오류 처리 검증
**테스트 단계**:
1. 유효한 거래처번호로 API 호출
2. startMonth를 endMonth보다 이후 날짜로 설정 (예: startMonth=2024-05, endMonth=2024-03)

**예상 결과**:
- 적절한 오류 상태 코드 반환 (400 Bad Request 등)
- 오류 메시지에 유효하지 않은 조회 기간에 대한 안내 포함

### 테스트 케이스 5: 필수 파라미터 누락 검증
**설명**: 필수 파라미터 누락 시 적절한 오류 처리 검증
**테스트 단계**:
1. 유효한 거래처번호로 API 호출
2. 필수 파라미터인 startMonth 또는 endMonth 누락

**예상 결과**:
- 상태 코드 400 Bad Request 반환
- 오류 메시지에 필수 파라미터 누락에 대한 안내 포함

### 테스트 케이스 6: 데이터가 없는 경우 검증
**설명**: 조회 기간 내 데이터가 없는 경우 응답 검증
**테스트 단계**:
1. 유효한 거래처번호로 API 호출
2. 데이터가 없는 기간으로 startMonth와 endMonth 설정

**예상 결과**:
- 상태 코드 200 OK 반환
- 빈 배열 또는 적절한 형식의 빈 데이터 반환

### 테스트 케이스 7: 대용량 데이터 조회 성능 검증
**설명**: 대용량 데이터 조회 시 API 성능 검증
**선행 조건**: 조회 기간 내 대량의 거래 데이터가 존재함
**테스트 단계**:
1. 대량의 거래 데이터가 있는 거래처번호로 API 호출
2. 충분히 긴 기간(예: 6개월)으로 startMonth와 endMonth 설정

**예상 결과**:
- 상태 코드 200 OK 반환
- 응답 시간이 허용 가능한 범위 내에 있음
- 모든 데이터가 정상적으로 조회됨

### 테스트 케이스 8: 다양한 결제 수단 포함 검증
**설명**: 다양한 결제 수단이 포함된 매출 상세 조회 기능 검증
**선행 조건**: 다양한 결제 수단으로 거래가 발생한 데이터가 존재함
**테스트 단계**:
1. 다양한 결제 수단으로 거래가 발생한 거래처번호로 API 호출
2. 해당 거래들이 포함된 기간으로 startMonth와 endMonth 설정

**예상 결과**:
- 상태 코드 200 OK 반환
- 응답 데이터에 다양한 결제 수단(신용카드, 배민 부담 가게배달팁 등)이 정확히 구분되어 표시됨
- 각 결제 수단별 금액 정보가 정확히 표시됨
