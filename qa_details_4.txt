# 부가세 참고자료 매출 집계 테스트 케이스

## 기본 정보
- **End Point**: `/api/v1/customer/{customerNo}/additional-tax/sales/summary`
- **Path Parameters**: customerNo (거래처번호)
- **Request Parameters**: startMonth, endMonth, excludeMeet

## 테스트 케이스

### 테스트 케이스 1: 기본 파라미터 검증
**목적**: 필수 파라미터 검증
- **입력**:
  - customerNo: 유효한 거래처번호
  - startMonth: 2024-01
  - endMonth: 2024-03
  - excludeMeet: 설정하지 않음(기본값 false)
- **예상 결과**: 
  - 상태 코드: 200 OK
  - 응답: 해당 기간의 부가세 참고자료 매출 집계 데이터 반환
- **검증 포인트**: 
  - 응답 데이터에 매출구분코드, 매출구분명, 공급가액, 부가세, 합계 필드가 포함되어 있는지 확인

### 테스트 케이스 2: 날짜 범위 검증
**목적**: 날짜 범위 파라미터 검증
- **입력**:
  - customerNo: 유효한 거래처번호
  - startMonth: 2024-03
  - endMonth: 2024-01 (종료일이 시작일보다 이전)
- **예상 결과**: 
  - 상태 코드: 400 Bad Request
  - 오류 메시지: 날짜 범위 오류 관련 메시지
- **검증 포인트**: 
  - API가 올바른 오류 응답을 반환하는지 확인

### 테스트 케이스 3: 배민 부담 가게배달팁 포함 확인
**목적**: 배민 부담 가게배달팁이 기타매출에 포함되는지 확인
- **입력**:
  - customerNo: 배민 부담 가게배달팁이 있는 거래처번호
  - startMonth: 2024-01
  - endMonth: 2024-03
  - excludeMeet: false
- **예상 결과**: 
  - 상태 코드: 200 OK
  - 응답: 기타매출에 배민 부담 가게배달팁 금액이 포함된 데이터
- **검증 포인트**: 
  - 응답의 salesType이 "PG"인 항목이 있는지 확인
  - salesTypeName이 "기타매출"인지 확인
  - 해당 항목의 금액에 배민 부담 가게배달팁 금액이 포함되어 있는지 확인

### 테스트 케이스 4: excludeMeet 파라미터 검증
**목적**: excludeMeet 파라미터가 만나서결제 제외에 영향을 주는지 확인
- **입력**:
  - customerNo: 만나서결제 내역이 있는 거래처번호
  - startMonth: 2024-01
  - endMonth: 2024-03
  - excludeMeet: true
- **예상 결과**: 
  - 상태 코드: 200 OK
  - 응답: 만나서결제 내역이 제외된 부가세 참고자료 매출 집계 데이터
- **검증 포인트**: 
  - 응답 데이터에 만나서결제 관련 항목이 제외되었는지 확인

### 테스트 케이스 5: 존재하지 않는 거래처번호 검증
**목적**: 존재하지 않는 거래처번호에 대한 처리 확인
- **입력**:
  - customerNo: 존재하지 않는 거래처번호
  - startMonth: 2024-01
  - endMonth: 2024-03
- **예상 결과**: 
  - 상태 코드: 404 Not Found 또는 적절한 오류 코드
  - 오류 메시지: 거래처를 찾을 수 없다는 내용
- **검증 포인트**: 
  - API가 존재하지 않는 거래처에 대해 적절한 오류 응답을 반환하는지 확인

### 테스트 케이스 6: 데이터가 없는 기간 검증
**목적**: 데이터가 없는 기간에 대한 응답 확인
- **입력**:
  - customerNo: 유효한 거래처번호
  - startMonth: 2020-01 (오래된 날짜로 데이터가 없을 것으로 예상)
  - endMonth: 2020-03
- **예상 결과**: 
  - 상태 코드: 200 OK
  - 응답: 빈 배열 또는 0값으로 채워진 데이터
- **검증 포인트**: 
  - 데이터가 없는 경우 API가 적절한 응답을 반환하는지 확인

### 테스트 케이스 7: 대용량 데이터 처리 검증
**목적**: 대용량 데이터 처리 성능 확인
- **입력**:
  - customerNo: 거래량이 많은 거래처번호
  - startMonth: 2023-01
  - endMonth: 2023-12 (1년 데이터)
- **예상 결과**: 
  - 상태 코드: 200 OK
  - 응답: 1년치 부가세 참고자료 매출 집계 데이터
- **검증 포인트**: 
  - API가 대용량 데이터를 적절한 시간 내에 처리하는지 확인
  - 응답 데이터의 정확성 확인

### 테스트 케이스 8: 잘못된 형식의 날짜 파라미터 검증
**목적**: 잘못된 형식의 날짜 파라미터 처리 확인
- **입력**:
  - customerNo: 유효한 거래처번호
  - startMonth: "2024/01" (잘못된 형식)
  - endMonth: 2024-03
- **예상 결과**: 
  - 상태 코드: 400 Bad Request
  - 오류 메시지: 날짜 형식 오류 관련 메시지
- **검증 포인트**: 
  - API가 잘못된 형식의 날짜에 대해 적절한 오류 응답을 반환하는지 확인

이 테스트 케이스들은 부가세 참고자료 매출 집계 API의 다양한 측면을 검증하며, 특히 배민 부담 가게배달팁이 기타매출에 올바르게 포함되는지 확인하는 데 중점을 두고 있습니다.
